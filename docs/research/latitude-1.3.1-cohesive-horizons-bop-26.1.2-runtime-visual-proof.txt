No source, build, tag, metadata, or biome JSON edits were made in this audit.

Evidence:
- Minecraft ChunkGenerator stores final biomeSource and memoizes featuresPerStep in the constructor from that constructor biomeSource plus generationSettingsGetter.
- ChunkGenerator.lambda$new$1 calls biomeSource.possibleBiomes(), List.copyOf(...), then FeatureSorter.buildFeaturesPerStep(...).
- applyBiomeDecoration later places from the precomputed featuresPerStep list, then invokes PlacedFeature.placeWithBiomeCheck.
- The default generationSettingsGetter uses Holder.value().getGenerationSettings(), so a real BOP biome holder would carry its BOP JSON features if that biome was included in the feature sorter input.
- LatitudeBiomeSource.collectPossibleBiomes returns original.possibleBiomes().stream(), not an augmented list of dynamically selectable Latitude replacements.
- ChunkGeneratorBiomeSourceMixin creates LatitudeBiomeSource from this.biomeSource.possibleBiomes() and only intercepts getBiomeSource(); it does not replace the constructor biomeSource field or rebuild ChunkGenerator.featuresPerStep.
- BOPOverworldRegionPrimary extends terrablender.api.Region, and seasonal_forest.json includes BOP vegetal features such as biomesoplenty:trees_seasonal_forest, flower_seasonal_forest, and maple leaf-litter placed features.

Audit answers:
1. Feature scheduling uses real biome generation settings, but only after the global feature sorter has been built from the generator constructor source's possibleBiomes.
2. Latitude is returning a real biomesoplenty:seasonal_forest holder for chunk biome identity; the mismatch is not just an ID/F3 illusion.
3. LatitudeBiomeSource does not advertise an augmented possible-biome set; it delegates collectPossibleBiomes to the original source.
4. BOP/TerraBlender provides seasonal_forest through Region integration and JSON features, but Latitude's tag-based replacement can return that biome outside the feature sorter's advertised source set.
5. The local 26.1.2 ChunkGenerator evidence supports a possible-biomes requirement: featuresPerStep is built from possibleBiomes, so dynamically returned replacement biomes whose features were not in that set will not schedule their placed features.

Best-supported mechanism:
BOP features are absent because Latitude can return biomesoplenty:seasonal_forest dynamically during biome lookup/population, but the feature scheduler's memoized featuresPerStep was built from the original biome source possible-biomes set before Latitude's dynamic BOP replacement was advertised. Result: many feature attempts occur in seasonal_forest biome positions, but the scheduled configured/placed feature list contains zero BOP configured features.

Updated classification:
BOP_OCCUPANCY_PROVEN_DECOR_MISMATCH_FEATURES_NOT_SCHEDULED_BY_POSSIBLE_BIOMES

   - For each PlacedFeature pf in that step: calls `indexedFeatures.indexMapping().applyAsInt(pf)` to get its sort index
   - Adds index to `intSet`
5. For each index in sorted `intSet`: calls `indexedFeatures2.features().get(index)` and places it.
