Optional-entry safety on 26.1.x
- Verdict: SAFE, same conclusion as the 1.20.1 branch-local audit
- Why:
  - 1.21.11-era Yarn still exposes optional tag entry APIs (TagEntry createOptional / createOptionalTag; 1.21.11 class-use and 1.21.8 TagEntry docs agree)
  - the 26.1 Latitude runtime resolver does not require every referenced ID to exist; it collects resolved holders from biomes.getTagOrEmpty(tag), sorts them, and returns base if the list is empty
  - the collection-backed path still filters by entry.is(tag), which naturally excludes unresolved optionals
- Practical meaning:
  - optional object-form biome refs should remain the safe way to carry absent-mod compat entries on the 26.1 line
  - raw string external IDs would still be the unsafe form if the mod is absent

Exact safe JSON shape for 26.1 planning
```json
{
  "replace": false,
  "values": [
    "minecraft:forest",
    { "id": "biomeswevegone:temperate_grove", "required": false },
```
