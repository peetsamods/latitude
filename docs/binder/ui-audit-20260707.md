## The two reported complaints, root-caused

**1. "The compass shifts around when the biome name is long."** CONFIRMED, one mechanism: the compass and
its attached text are measured as ONE box, and the anchor position is computed FROM the box's width.
`computeAnalogBounds` folds `font.width(zoneText)` into `boxW` (`CompassHud.java:559-584`, text width at
:564/:570); `anchoredX` then places the box: `CENTER -> (screenW - scaledBoxW) / 2`,
`RIGHT -> screenW - scaledBoxW - 4` (`CompassHud.java:721-726`); the analog disc's center is `x + radius`
(:318). Longer biome ⇒ wider box ⇒ new anchored x ⇒ the disc itself slides (half the width delta at
CENTER, the full delta at RIGHT). The digital style has the identical coupling (`maxLineWidth` → boxW →
`anchoredX`, :201/:224) — direction, coords, zone, and biome are concatenated into one measured line
(`buildDigitalLine`). **The shipped default config is exactly the worst case**: `style=ANALOG` +
`hAnchor=CENTER` (`CompassHudConfig.java:32/37`). Every variable-width source feeds the same box: biome
name, zone word (Tropics→Subtropics), lat/lon readout (worse since longitude shipped), the direction
token, and the compact-separator choice. Detached elements are decoupled from the compass but a
CENTER/RIGHT-anchored detached label still re-centers around its own text.

**2. "Freeform placement in the Studio is misleading vs in-game."** CONFIRMED — seven concrete
divergences; A+B+C fully explain wrong coordinates, D-G mislead about rules/appearance:
- **A.** Preview and hitboxes measure FIXED sample text ("Plains" :860, "Tropics" :490, "1°S, 15°E" :475)
  while live measures real variable-width names (:851/:827/:494).
- **B.** A drag stores `offset = target − computeBasePosition(...)` (`LatitudeHudStudioScreen.java:742-744`)
  where the base position was anchored using the SAMPLE width (`CompassHud.java:162/164`) — live
  re-anchors with the real width, so the element lands where the Studio never showed it.
- **C.** The stored offset is an opaque anchor-relative delta (depends on anchor + text width + screen
  size) — not the top-left or center of any fixed thing.
- **D.** Two placement models in one editor: compass drag stores top-left-relative grab (:1214), title
  drag stores center-relative (:664-669/:711-714).
- **E.** Preview draws a fake hotbar + checkerboard the live HUD lacks (:650/:376).
- **F.** Preview shows all elements unconditionally; live hides everything unless `showMode` is satisfied
  (`CompassHud.java:105`) — you can lovingly place an in-game-invisible HUD.
- **G.** Preview draws borders around elements that render borderless live.

## Hidden problems found (beyond the reported two)

- **Placement does not survive GUI-scale changes or resizes**: every stored offset is absolute
  GUI-scaled pixels (`CompassHud.java:531-532` etc.); CENTER/RIGHT/BOTTOM anchors shift with resolution;
  nothing re-normalizes. Clamping is render-time only (:581/:964/:1037) — out-of-range offsets persist
  invisibly, and a longer live biome name can clamp an element to a different resting spot than shown.
- **Hot-path waste**: the analog disc is drawn with one `ctx.fill` PER PIXEL (:392-407 — ~4k-16k quads
  per frame at max size); per-frame `Component.literal`/`StringBuilder` allocation (:303/:518/:926); the
  biome registry lookup re-runs every frame uncached (:855).
- **Stale state across world/dimension switches**: `ZoneEnterTitleOverlay` keys on
  `client.level.getGameTime()` and never clears its statics on unload (:30-31/:50) — stale or
  early-vanishing titles; `CompassHud.cachedHasCompass` one frame stale post-switch (:93-96).
- **Default position collides with the vanilla boss bar** (TOP-CENTER).
- **Title drag hitbox is hardcoded to measure `"TROPICAL 0°"`** (`LatitudeHudStudioScreen.java:1225`) —
  the grab box drifts from long/styled titles (ignores letter-spacing/case too).
- **F9 opens the legacy `LatitudeSettingsScreen`, not the Studio** (`GlobeModClient.java:179-185`); the
  Studio is button-only (via Settings :82 and the create screen :1066). Five fields are editable in BOTH
  screens; the Settings side of `zoneEnterTitleEnabled`/`displayZoneInHud` doesn't `saveCurrent()` (Esc
  loses the edit; the Studio side saves), and `compactHud` is shown Digital-only in the Studio (:342) but
  unconditionally in Settings (:144) — same field, different visibility semantics.
- **Config model**: two schemaless GSON files, no version field, no migration story, no exit-time save;
  `LatitudeConfig` hand-mirrors static/`…Value` pairs (each field written ~4×) with THREE hardcoded
  default sites and no `fresh()` — the drift class that already bit compass resets once.
- **Dead config fields** (zero consumers, grep-confirmed): `zoneEntryNotifyMode`, `showLatitudeDegrees`,
  `latitudeBandBlend*` trio, `enableEwStormWall`, `debugLatitudeBlend`, plus TWO near-identically-named
  dead compass-degree flags. One live knob with NO UI: `CompassHudConfig.directionMode`.
- **Seed-0 has no UI guard**: `seedField.getValue()` flows raw to `wc.setSeed`
  (`LatitudeCreateWorldScreen.java:1075`, `LatitudeWorldLauncher.java:108`); a literal typed `0` makes the
  mod's whole geography permanently inert (worldgen audit P0-2 class) with only a server-log line
  (Slice B's warn) to say so. The create screen is the right place to catch it.
- **`GlobeWorldSize.label` block counts are stale Classic squares** ("20,000 x 20,000"); the screen
  papers over it with its own honest Mercator-aware `worldDimsLabel()` (:2100) — the enum text is a
  latent trap for any other consumer. The UI-name↔preset-id mismatch (UI Small→`globe_regular`, UI
  Regular→`globe_large`, `GlobeWorldSize.java:10-12`) is player-invisible but a standing dev/log/recipe
  hazard — worth a launch-time log of `worldPresetId`.
- **Dead classes still shipping** (all zero-caller, confirmed): `OverlayProof` (fake "PROOF OVERLAY
  LIVE"), `ZoneEntryNotifier` + `ui/ZoneTitleOverlay` (the live path is `ZoneEnterTitleOverlay`),
  `EwSandstormOverlayRenderer` (no-op stub + stale import in `GlobeModClient:15`), `GlobeModMenu` (empty
  stub — and no ModMenu entry exists at all), the old circular `LatitudePlanisphereRenderer.render()`.
  Loading overlay nit: `globe$displayProgress` is written but never rendered. `SpawnZoneScreen` is live
  but vanilla-look legacy.

## What genuinely works (keep, and build on)

- **Create screen**: mature derived-geometry layout with a reasoned three-column↔tabbed responsive
  fallback (`COMFORTABLE_THREE_COL_W=530`), per-pane smooth scroll with edge clipping, honest
  Mercator-aware dimension labels, faithful launcher (every selection forwarded, pending-state rollback,
  bonus-chest fix). The 06-22 rejection history stays respected: refinement only, and that's all it needs.
- **Loading overlay**: honest (progress strictly bounded by vanilla's real `smoothedProgress`), robust
  render gate (settle + 3×3 ring + warmup + graduated 6s/15s/10min timeouts), `onClose` suppressed while
  loading. Current with 26.2 APIs.
- **Studio machinery**: preview renders through the REAL `CompassHud` draw path (geometry cannot drift —
  only its INPUTS lie, which is exactly why the fix is cheap); tabs rebuild cleanly; drag persistence is
  release-gated (no per-frame writes); reusable RGB-picker/swatch component; `CompassHudConfig.fresh()`
  single-source defaults + `sanitize()` resilience; append-only enum discipline.
- Centralized rainbow palette (`RainbowText`); radius-scaled analog glyphs/ticks; clean attached/detached
  text mirroring (`joinOrdered`).

## Verdict

The bones are good — one wrong layout primitive (content-width-coupled anchoring) sits under both
reported complaints and most of the hidden placement problems, the Studio's preview inputs (not its
renderer) cause the mismatch, and the remaining findings are hygiene (IA split, config model, dead code,
hot-path waste, seed-0 guard). The companion design fixes the primitive once and everything above it
inherits the fix.
