## Scope landed (Stage 1 / Bucket 1 — worldgen logic only)

This is an **algorithm transplant**, not a cherry-pick: the 1.4 source (26.1.2, Mojang mappings) was authored on a
**province/moisture** worldgen architecture that **does not exist in the 1.3.0 line** (no `ProvinceAuthority`,
no `Province` enum, no `enforceWarmProvinceFamily`, no moisture classifier). The pure-logic 1.4 changes were therefore
re-grounded onto the target's existing Yarn (`RegistryEntry<Biome>`) worldgen and validated map-based.

**Porting fact:** the 1.20.1 baseline `LatitudeBiomes.java` differs from the 1.21.x baseline by ~695 lines (older API,
reordered/extra helpers), so the transplant was applied **surgically** (5 targeted hunks) rather than by file copy.
Every helper symbol the 1.4 logic depends on (`isBadlandsFamily`, `isBiomeId`, `biome(...)`, `entryById`, `isSavannaFamily`,
`setSelectionPath`, `PATH_FALLBACK_PICK`, `ACTIVE_RADIUS_BLOCKS`, `REFERENCE_DIAMETER_BLOCKS`, `BAND_POLAR`, `WORLD_SEED`,
`ValueNoise2D.sampleBlocks`, `smoothstep`, `savannaTierByY`, `pick.getKey()...`) is present in this baseline with the same
signatures, and the five anchor sites (`isBadlandsFamily` block, the `VARIANT_CELL_SIZE_BLOCKS` constant block, the
`pickFrom` body, and the polar/savanna regions of **both** `sanitizeLandBiome`/`applyFinalSavannaClimateClamp` overloads)
are textually identical to the 1.21.x baseline. The resulting diff is `+152 / -7` — byte-for-byte the same transplant
shape that landed on 1.21.11 and 1.21.1, confirming the worldgen algorithm is mapping-independent.
